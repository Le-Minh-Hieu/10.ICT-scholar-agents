# ICT-SCHOLAR-AGENTS-V1: Macro Cognition Generation Layer Audit

This audit focuses exclusively on the generation of the macro context, weekly profile building, narrative synthesis, calendar parsing, and retrieval mechanisms.

---

## 1. Weekly Profile Generation Flow

```
[Calendar Events]
       | (MacroCalendarStore.load or getLatestWeek)
       v
[Classification]
       | - Deterministic week_type matching (FOMC, CPI, NFP, GENERIC)
       | - Volatility expectation heuristics (based on max impact score)
       v
[Base Profile Structure]
       |
       v
[Retrieval Adapter]
       | - Construct context-aware queries (drivers, volatility, events)
       | - retrieveRAG vector database search
       v
[Event-Level Grounded Reasoning]
       | - LLM-based reasonAboutNews calls for active/upcoming events
       v
[Narrative Synthesis & Adaptation]
       | - heuristic token matching (bullish/bearish counts)
       | - price validation & regime adaptation
       v
[MacroContextState] (Persisted Weekly Profile)
```

### Flow Artifacts & Intermediate Objects

1. **Input Parameters**:
   * `weekStartIso`: Target week date (ISO string, e.g. `"2026-05-25"`).
   * `opts.pmso`: (Optional) Current master state for query refinement.
   * `opts.priceSnapshot`: (Optional) Current price snapshot for deviation testing.

2. **Loaded Weekly Calendar State (`weekState`)**:
   * Loaded from `data/calendar_cache/` matching `weekStartIso`. Contains raw events array.

3. **Base Profile Skeleton (`baseProfile`)**:
   * Initialized prior to retrieval:
     ```javascript
     {
       week_start: weekStart,
       week_type: "GENERIC" | "FOMC" | "CPI" | "NFP",
       primary_drivers: string[],
       volatility_expectation: "LOW" | "MEDIUM" | "HIGH",
       delivery_model: 'mixed',
       macro_bias: 'neutral',
       narrative_confidence: 0,
       active_events: [...],
       upcoming_events: [...],
       retrieval_context: { expandedQueries: [] },
       narrative_state: '',
       regime: { volatility: 'MEDIUM', liquidity: 'STABLE', macro_alignment: 'NEUTRAL' },
       narrative_history: []
     }
     ```

4. **Retrieval Package (`retrieval`)**:
   * Output of `retrieveForMacroProfile(baseProfile)`. Contains:
     * `queries`: Array of string queries used in vector search.
     * `rag`: Raw database results containing `chunks[]`.

5. **Reasoning Results (`reasoningResults`)**:
   * Array of individual `NewsReasoningResult` objects produced by looping candidates through `reasonAboutNews(event, rag.chunks)`.

6. **Synthesized output (`enriched`)**:
   * The final mutated `MacroContextState` containing updated `macro_bias`, `narrative_state`, `narrative_confidence`, `regime` parameters, and `narrative_history` entry.

## 2. Macro Bias Generation Audit

The system produces `macro_bias` using **heuristic token matching** on the top retrieved text chunks.

### Generation Steps
```
                    +------------------------------------+
                    |        RAG Retrieval Chunks        |
                    +------------------------------------+
                                      |
                                      v
                    +------------------------------------+
                    |  Extract top 30 chunk text blocks  |
                    +------------------------------------+
                                      |
                                      v
                    +------------------------------------+
                    | Scan lowercase text for key tokens  |
                    +------------------------------------+
                                 /          \
      ("bull", "strength", "discount")     ("bear", "weak", "premium")
                               /              \
                              v                v
                       bullish += 1        bearish += 1
                               \              /
                                v            v
                    +------------------------------------+
                    |  Determine dominant count direction|
                    +------------------------------------+
```

### Deterministic & Heuristic Paths

* **Deterministic Text Mapping**:
  * Case-insensitive matching maps occurrences of exactly these token words:
    * **Bullish Tokens**: `bull`, `strength`, `discount`
    * **Bearish Tokens**: `bear`, `weak`, `premium`
* **Heuristic Scoring**:
  * If `bullishCount > bearishCount` → `macro_bias = "bullish"`
  * If `bearishCount > bullishCount` → `macro_bias = "bearish"`
  * Otherwise → `macro_bias = "neutral"`
* **LLM Synthesis**:
  * **None**: The weekly profile's overall `macro_bias` does **NOT** use LLM synthesis. The event-level `NewsReasoningResult` values generated by `reasonAboutNews` use an LLM, but their aggregate results are not parsed by the narrative engine to decide `macro_bias`.
* **Retrieval Evidence**:
  * Chunks are fetched via vector search utilizing queries formatted in `macro-retrieval-adapter.ts`.

### Confidence Calculation
Computed in `macro-narrative-engine.ts:L29-30`:
1. **Average Uncertainty**: `avgUncertainty` is computed as the average `uncertainty_pressure` across all event reasoning results.
2. **Bias Divergence Boost**:
   $$\text{boost} = \min\left(0.3, \frac{|\text{bullishCount} - \text{bearishCount}|}{10}\right)$$
3. **Narrative Confidence**:
   $$\text{narrative\_confidence} = \max\left(0, \min\left(1, 1 - \text{avgUncertainty} \times 0.9 + \text{boost}\right)\right)$$

## 3. Narrative State Generation Audit

The `narrative_state` is a **static heuristic template** combining deterministic week parameters.

### Generation Flow
1. **Source Events**: Selects `active_events` (events in `PRE_EVENT`, `POST_EVENT`, or `COOLDOWN` window phase). Falls back to the first 3 `upcoming_events` if no active events exist.
2. **Retrieval Chunks**: Evaluates RAG chunks to assign the `macro_bias` and `regime.volatility` fields.
3. **Synthesis Prompt**:
   * **None**: The system does **not** call an LLM to write the weekly `narrative_state`.
   * **Template Equation**:
     $$\text{narrative\_state} = \text{"Week } \text{week\_start} \text{: } \text{week\_type} \text{ — bias=} \text{macro\_bias} \text{; expect } \text{regime.volatility.toLowerCase()} \text{ volatility."}$$
4. **Fallback Behavior**:
   * If RAG returns no matching chunks, `macro_bias` defaults to `"neutral"`, and `regime.volatility` stays `"MEDIUM"`, yielding:
     `"Week <weekStart>: GENERIC — bias=neutral; expect medium volatility."`

## 4. Event Validity & Classification Audit

### Classification Logic
Inside `weekly-profile-builder.ts`, `week_type` is determined using a simple string containment search:
```typescript
let week_type = 'GENERIC';
for (const ev of events) {
  const name = String(ev.name || ev.title || '').toUpperCase();
  if (name.includes('FOMC')) week_type = 'FOMC';
  if (name.includes('CPI')) week_type = 'CPI';
  if (name.includes('NFP')) week_type = 'NFP';
}
```

### Vulnerability Analysis: Over-merging & Incorrect Classification

#### 1. Over-Classification Risk
Since the check scans for the string `"FOMC"` anywhere in the name, the system will classify a week as an `"FOMC"` week if it contains:
* Minor events: `"FOMC Member Speech"`, `"Fed Governor speech at FOMC Luncheon"`.
* Retrospective/Delayed releases: `"FOMC Minutes"`.
* Major meetings: `"FOMC Interest Rate Decision"`.

This creates a high risk of over-classifying generic weeks as high-priority `"FOMC"` weeks, misrepresenting the volatility regime.

#### 2. Under-Classification Risk
A week featuring an critical interest rate announcement or testimony by the Chairman that is titled `"Fed Speech"` or `"Fed Testimony"` without containing the exact letters `"FOMC"` will bypass the FOMC classification entirely and fall back to `"GENERIC"`.

#### 3. Event Merging & Deduplication (`dedupe.ts`)
* Events are deduped using `stableIdForEvent`:
  $$\text{Stable ID Key} = \text{currency} \mid \text{category} \mid \text{scheduled\_time} \mid \text{normalized\_title}$$
* Under this scheme, `"FOMC Meeting"` and `"FOMC Minutes"` are treated as **distinct** events because their normalized titles differ.
* They are **not** merged into a single event. However, their co-existence in the weekly schedule means the loop will encounter the word `"FOMC"` multiple times, repeatedly confirming `"FOMC"` as the `week_type`.

## 5. Retrieval Context Audit

`retrieval_context` generation is managed by `macro-retrieval-adapter.ts`:

```
                 +---------------------------------------+
                 |            Input Profile              |
                 +---------------------------------------+
                                     |
                                     v
                 +---------------------------------------+
                 |      Query Construction Pipeline       |
                 | - "week_type macro week"              |
                 | - "driver macro driver" (per driver)  |
                 | - "volatility regime.volatility"      |
                 | - First sentence of narrative_state   |
                 | - "title/id" (per active event)       |
                 +---------------------------------------+
                                     |
                                     v
                 +---------------------------------------+
                 |    Vector RetrieveRAG Search Call     |
                 +---------------------------------------+
                                     |
                                     v
                 +---------------------------------------+
                 |      Ranking, Slicing & Storage       |
                 | - Vector database returns similarity  |
                 | - Top 10 IDs saved in top_chunks     |
                 | - Top 30 scanned for bias count       |
                 +---------------------------------------+
```

1. **Retrieval Source**: Vector database accessed through the `retrieveRAG` service core.
2. **Query Construction**:
   * Adds: `` `${profile.week_type} macro week` ``
   * Adds: `` `${driver} macro driver` `` for up to 6 primary drivers.
   * Adds: `` `volatility ${profile.regime.volatility}` ``
   * Adds: First sentence of `profile.narrative_state` (if present).
   * Adds: `` `${event.title || event.id}` `` for all active events.
   * Baseline Fallback: If no queries are generated, defaults to `"macroeconomic drivers this week"`.
3. **Ranking**: Determined entirely by the vector similarity scoring against the search queries.
4. **Filtering**: None. No time boundaries, currency restrictions, or category exclusions are applied within the retrieval adapter.
5. **Evidence Selection**: Slices the top 10 ranked chunk IDs to populate `profile.retrieval_context.top_chunks`, and evaluates the top 30 chunk texts for `macro_bias` token counts.

## 6. Failure Modes Analysis

### A. Wrong `macro_bias`
* **Cause**: Context-free heuristic word-matching.
* **Concrete Example**: The system retrieves a chunk regarding the US Dollar: *"US Dollar index (DXY) is extremely bullish, pushing treasury yields up"*. Because `"bullish"` is parsed, `bullishCount` increases. However, a bullish DXY implies a **bearish** EURUSD. The naive count interprets this as a *bullish* macro bias for EURUSD, generating a false signal.

### B. Over-Generalized `narrative_state`
* **Cause**: Lack of LLM synthesis in the narrative generation step.
* **Concrete Example**: A complex week featuring conflicting CPI data (high inflation) and rising unemployment is summarized simply as: `"Week 2026-05-25T00:00:00Z: CPI — bias=bullish; expect high volatility."` The nuanced tension between inflation and employment is completely lost.

### C. Incorrect `week_type`
* **Cause**: Naive string containment search in event titles.
* **Concrete Example**: A holiday week contains no major events, except for a minor speech by a Fed member titled `"Fed member speech at FOMC dinner"`. Because the string `"FOMC"` is found, the system classifies the week as `"FOMC"` rather than `"GENERIC"`, triggering high-volatility pre-event and embargo constraints on unrelated trading days.

### D. Incorrect `volatility_expectation`
* **Cause**: reliance on ForexFactory impact tags.
* **Concrete Example**: A bank holiday in Europe severely drains liquidity, causing sharp, erratic price expansions. Since the holiday event is tagged `"LOW"` impact in the calendar, `volatility_expectation` is computed as `"LOW"`, and the system fails to account for the liquidity strain.

## 7. Deliverable: Architecture Audit Matrix

| Artifact | Generated By | Inputs | Failure Modes |
| :--- | :--- | :--- | :--- |
| **`week_type`** | `weekly-profile-builder.ts` | Title/name strings of all weekly events | * **Over-Classification**: Minor speeches containing `"FOMC"` classify the week as `"FOMC"`. <br> * **Under-Classification**: Speeches or testimonies by the Fed chair titled `"Fed Testimony"` (omitting `"FOMC"`) fall back to `"GENERIC"`. |
| **`macro_bias`** | `macro-narrative-engine.ts` | Top 30 text chunks from `retrieveRAG` | * **Context Inversion**: Positive words ("bullish") referencing inverse assets (e.g. DXY) are incorrectly registered as bullish for the primary asset (EURUSD). <br> * **Noisy Retrieval**: Irrelevant documents containing words like `"weak"` distort the count. |
| **`narrative_state`** | `macro-narrative-engine.ts` | `week_start`, `week_type`, `macro_bias`, `regime.volatility` | * **Zero Nuance**: Static string template misses all economic complexity, data conflicts, and central bank developments. |
| **`volatility_expectation`** | `weekly-profile-builder.ts` | `impact` tags of all weekly events | * **Impact Rating Errors**: Relies on high-impact designations; misses systemic volatility catalysts (e.g., low-liquidity bank holidays). |
| **`active_events`** | `weekly-profile-builder.ts` | `events` array, `lifecycle_phase` values | * **Stale Phase Data**: If lifecycle states are not refreshed in staging, active events remain stuck in past window states. |
| **`retrieval_context`** | `macro-retrieval-adapter.ts` | `week_type`, `primary_drivers`, `regime`, `narrative_state`, `active_events` | * **Diluted Search**: Constructing up to 20 raw string queries dilutes vector search precision, returning low-relevance results. <br> * **No Date/Currency Filters**: Retrieves historic documents regardless of timeline or currency relevance. |
